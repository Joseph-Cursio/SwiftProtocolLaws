# SwiftPropertyLaws v1.9.0

**Released:** 2026-05-02
**Tag:** `v1.9.0`
**Previous release:** [v1.8.0](https://github.com/Joseph-Cursio/SwiftPropertyLaws/releases/tag/v1.8.0)

## TL;DR

Second **kit-defined** protocol cluster, layered on top of v1.8.0's Semigroup + Monoid. Adds `CommutativeMonoid`, `Group`, and `Semilattice` to `PropertyLawKit` so SwiftInferProperties' M8 algebraic-structure-composition pass can write conformances the kit verifies on every CI run for the wider PRD §5.4 promotion table — single-Monoid → CommutativeMonoid (commutativity), Monoid + inverse → Group, CommutativeMonoid + idempotence → Semilattice. Three new protocols, four new Strict-tier own laws (one CMon, two Group, one Semilattice), zero changes required in `PropertyLawSuiteMacro` or `PropertyLawDiscoveryTool`.

## What's new

### `CommutativeMonoid` *(kit-defined, in `PropertyLawKit`)*

```swift
public protocol CommutativeMonoid: Monoid {}
```

One Strict-tier own law plus the inherited Monoid + Semigroup chain via `.all`:

- `CommutativeMonoid.combineCommutativity` — `combine(a, b) == combine(b, a)`

No new requirements — pure marker protocol that triggers the commutativity law check on top of the inherited Monoid surface. Verified by `checkCommutativeMonoidPropertyLaws(..., laws: LawSelection = .all)`. `.ownOnly` skips the inherited Monoid + Semigroup checks.

### `Group` *(kit-defined, in `PropertyLawKit`)*

```swift
public protocol Group: Monoid {
    static func inverse(_ x: Self) -> Self
}
```

Two Strict-tier own laws plus the inherited Monoid + Semigroup chain via `.all`:

- `Group.combineLeftInverse` — `combine(inverse(x), x) == .identity`
- `Group.combineRightInverse` — `combine(x, inverse(x)) == .identity`

Verified by `checkGroupPropertyLaws(..., laws: LawSelection = .all)`. The `inverse(_:)` static-method form (rather than `inverted()` instance method or `negated()` overlap with `SignedNumeric.negate()`) keeps the witness-extraction shape uniform with `combine(_:_:)` and `identity`. SwiftInferProperties' M8 RefactorBridge aliases user-named inverse witnesses (`negate`, `reciprocal`, `inverted`, `complement`) into `Self.inverse(_:)` via the same one-line static-aliasing pattern v1.8 established for `combine` / `identity`.

### `Semilattice` *(kit-defined, in `PropertyLawKit`)*

```swift
public protocol Semilattice: CommutativeMonoid {}
```

One Strict-tier own law plus the inherited CommutativeMonoid + Monoid + Semigroup chain via `.all`:

- `Semilattice.combineIdempotence` — `combine(a, a) == a`

No new requirements — pure marker protocol that triggers the idempotence law check on top of the full CommutativeMonoid surface. Bounded join-semilattices (`(Set<T>, ∪, ∅)`, `(Int, max, .min)`) and bounded meet-semilattices (`(Int, min, .max)`, `(Bool, &&, true)`) both use the same `Semilattice` conformance — the law is symmetric.

### Inheritance shape (v1.9 cluster + v1.8 base)

```
Semigroup
  └─ Monoid
       ├─ CommutativeMonoid
       │    └─ Semilattice
       └─ Group
```

`Group` and `CommutativeMonoid` are **incomparable** — non-commutative groups are valid (matrix multiplication groups, permutation groups), and `CommutativeGroup` is intentionally not in v1.9 scope (rare in idiomatic Swift; defer until corpora justify it). A type that's both a `CommutativeMonoid` and a `Group` (e.g., the canonical additive-integer Abelian group) emits **both** law checks under macro / discovery dispatch. SwiftInferProperties M8 plan open decision #6 default `(a)` matches this — emit both proposals, with a v1.1+ kit-side `CommutativeGroup` as the natural future collapse.

### `KnownProtocol` — adds `.commutativeMonoid` / `.group` / `.semilattice`

`PropertyLawCore.KnownProtocol` gains three cases plus their `typeNameMap`, `subsumedProtocols`, `checkFunctionName`, `declarationName`, and `testNameFragment` switch arms.

`subsumedProtocols`:
- `.commutativeMonoid → [.monoid, .semigroup]`
- `.group → [.monoid, .semigroup]` *(does NOT subsume CommutativeMonoid — see incomparability above)*
- `.semilattice → [.commutativeMonoid, .monoid, .semigroup]`

`PropertyLawSuiteMacro` and `PropertyLawDiscoveryTool.GeneratedFileEmitter` required **zero source changes** — both consume `KnownProtocol` generically and pick the new cases up automatically. v1.9.4's integration tests confirm this end-to-end: macro expansion of `: CommutativeMonoid, Group` emits both `checkCommutativeMonoidPropertyLaws` and `checkGroupPropertyLaws` calls; the discovery scanner walking real source files reduces inheritance clauses to the most-specific KnownProtocol set per `mostSpecific`.

### Negative-control coverage

Per PRD §8 framework self-test gate, each new law gets a planted-bug fixture that confirms the violation fires:

- **`NonCommutativeString`** — string concat is associative + has identity but is non-commutative; surfaces the `CommutativeMonoid.combineCommutativity` violation.
- **`BrokenInverse`** — returns `x` for `inverse(x)` (identity instead of inverse); surfaces both `Group.combineLeftInverse` and `Group.combineRightInverse` violations. Generator skips 0 so every sample exposes the bug.
- **`NonIdempotentTally`** — integer addition is a CommutativeMonoid but not idempotent (`1 + 1 ≠ 1`); surfaces the `Semilattice.combineIdempotence` violation. Generator skips 0 for the same reason.

## Why this release exists

SwiftInferProperties v0.4 PRD §5.4 describes the algebraic-structure composition table:

| Detected combination | Implies | RefactorBridge target |
|---|---|---|
| binary op `(T, T) → T` + associativity | Semigroup | kit-defined Semigroup *(v1.8)* |
| Semigroup + identity element | Monoid | kit-defined Monoid *(v1.8)* |
| Semigroup + commutativity + identity | CommutativeMonoid | **kit-defined CommutativeMonoid *(v1.9)*** |
| Monoid + inverse | Group | **kit-defined Group *(v1.9)*** |
| CommutativeMonoid + idempotence | Semilattice | **kit-defined Semilattice *(v1.9)*** |
| Two compatible structures, distributive | Ring | stdlib `Numeric` *(unchanged in v1.9)* |
| `T → T` self-functions with inverse + identity | Group acting on T | informational *(no kit/stdlib protocol)* |

Until v1.9, the middle three rows had no kit/stdlib protocol target — SwiftInferProperties' M8 RefactorBridge would have written claim-only comment-block files for them. v1.9 closes the loop for those three so M8.5 can emit real `extension TypeName: <Protocol> {}` writeouts the kit verifies on every CI run.

`Ring` and `CommutativeGroup` are intentionally deferred — Ring needs two-op shape design (additive group + multiplicative monoid + distributivity laws) the kit's single-op `static func combine(_:_:)` pattern doesn't cleanly express; CommutativeGroup is mathematically clean but rare enough in idiomatic Swift to defer until real corpora flag it. Both are reasonable v1.10 / v1.11 candidates if SwiftInferProperties' calibration data motivates them.

## Upgrade path

From v1.8.0:

1. **No source changes for existing consumers.** v1.9 is purely additive — Semigroup / Monoid + every prior cluster compile unchanged.
2. **New downstream consumers** writing CommutativeMonoid:
   ```swift
   import PropertyLawKit

   struct Tally: CommutativeMonoid, Equatable {
       var count: Int
       static let identity = Tally(count: 0)
       static func combine(_ lhs: Tally, _ rhs: Tally) -> Tally {
           Tally(count: lhs.count + rhs.count)
       }
   }

   try await checkCommutativeMonoidPropertyLaws(for: Tally.self, using: Tally.gen())
   ```
3. **Group** adds the `inverse(_:)` requirement:
   ```swift
   struct AdditiveInt: Group, Equatable {
       let value: Int
       static let identity = AdditiveInt(value: 0)
       static func combine(_ lhs: AdditiveInt, _ rhs: AdditiveInt) -> AdditiveInt {
           AdditiveInt(value: lhs.value + rhs.value)
       }
       static func inverse(_ x: AdditiveInt) -> AdditiveInt {
           AdditiveInt(value: -x.value)
       }
   }
   ```
4. **Semilattice** is the same shape as CommutativeMonoid (no new requirements):
   ```swift
   struct MaxInt: Semilattice, Equatable {
       let value: Int
       static let identity = MaxInt(value: .min)
       static func combine(_ lhs: MaxInt, _ rhs: MaxInt) -> MaxInt {
           MaxInt(value: max(lhs.value, rhs.value))
       }
   }
   ```
5. **`@PropertyLawSuite`-marked types** with `: CommutativeMonoid` / `: Group` / `: Semilattice` in their inheritance clause now generate the corresponding `check…` calls automatically.
6. **`swift package propertylawcheck discover`** picks up the new conformances in scanned targets the same way it picks up stdlib + v1.8 protocols.

## What's deliberately not in this release

- **`Ring`.** Two-op shape (additive group + multiplicative monoid + distributivity) doesn't fit the kit's single-op `combine` / `identity` pattern. Designing a dual-witness or layered `AdditiveGroup` + `MultiplicativeMonoid` + `Ring` stack is reasonable v1.10+ work; for now, SwiftInferProperties M8 still targets stdlib `Numeric` for the Ring promotion (PRD §5.4 row 5), gated on TypeShape numeric-shape detection.
- **`CommutativeGroup`.** Mathematically the natural collapse for types satisfying both `CommutativeMonoid` and `Group`, but rare in idiomatic Swift (most everyday Abelian groups are integer / floating-point under `+`, already covered by stdlib `AdditiveArithmetic` / `Numeric`). SwiftInferProperties M8.4's incomparable-arm split (open decision #6) emits both proposals; v1.10+ kit-side CommutativeGroup would collapse them.
- **`Group acting on T`.** Function-space carrier (permutation groups, transformations on a set) doesn't fit the kit's per-type protocol shape — the carrier is the type's *automorphism* space, not the type itself. PRD §5.4 row 6 stays informational. Defer indefinitely unless a strong use case emerges.
- **Operator sugar.** `static func combine(_:_:)` stays canonical; `inverse(_:)` stays a static method. No `infix operator <>` / `prefix operator ~`. Same posture as v1.8.
- **`DerivationStrategist` arms for the v1.9 conformances.** Conforming to CommutativeMonoid / Group / Semilattice doesn't help generate values of the type. The `.derived` priority list (PRD §5.7) stays as-is.
- **Validation-pass coverage of the new laws.** Same posture as v1.8 — no real-world Swift packages with `: CommutativeMonoid` / `: Group` / `: Semilattice` shapes exist yet (the protocols just shipped). Defer to post-1.9 adoption metrics; `Validation/FINDINGS.md` gets new entries once downstream users adopt.
- **Performance budget re-validation.** CommutativeMonoid commutativity is 2-sample / 1-property; Group inverse laws are 1-sample / 1-property each; Semilattice idempotence is 1-sample / 1-property. Same shape as the existing Monoid identity laws — no new perf characterization needed.
