# SwiftProtocolLaws v1.6.0

**Released:** 2026-04-30
**Tag:** `v1.6.0`
**Previous release:** [v1.5.0](https://github.com/Joseph-Cursio/SwiftProtocolLaws/releases/tag/v1.5.0)

## TL;DR

Single-purpose release: exposes `ProtoLawCore` as a public library product so the SwiftInferProperties downstream package can call into the shared `DerivationStrategist`. Resolves the M3 prerequisite the SwiftInferProperties PRD has been blocking on (PRD §11, §21 OQ #4). No other behaviour changes.

## What's new

### `ProtoLawCore` is now a shipped library product

Added to `Package.swift`:

```swift
.library(
    name: "ProtoLawCore",
    targets: ["ProtoLawCore"]
)
```

Downstream packages can now `import ProtoLawCore` and call into the §5.7 generator-derivation strategist:

```swift
import ProtoLawCore

let shape = TypeShape(
    name: "MyType",
    kind: .struct,
    inheritedTypes: ["Equatable"],
    hasUserGen: false,
    storedMembers: [StoredMember(name: "value", typeName: "Int")]
)
let strategy: DerivationStrategy = DerivationStrategist.strategy(for: shape)
```

### Promoted to `public` in `Sources/ProtoLawCore/DerivationStrategy.swift`

| Symbol | Purpose |
|---|---|
| `DerivationStrategy` (enum + cases) | Strategist output: `.userGen`, `.caseIterable`, `.memberwiseArbitrary`, `.rawRepresentable`, `.todo` |
| `MemberSpec` | Per-member derivation spec (name + raw type) |
| `RawType` (enum + cases + `init?(typeName:)` + `generatorExpression`) | Recognized stdlib raw types for `RawRepresentable` derivation |
| `StoredMember` | Stored-property record (name + textual type spelling) |
| `TypeShape` (struct + nested `Kind` enum) | Strategist input: type kind, inheritance, members, user-init flag |
| `DerivationStrategist` (enum + `memberwiseArityLimit` + `strategy(for:)`) | Pure-logic strategist |

### Stayed `package`-scoped

| File | Why |
|---|---|
| `KnownProtocol.swift` | Internal stdlib-protocol lookup table consumed only by `ProtoLawMacroImpl` and `ProtoLawDiscoveryTool`; not part of the contract SwiftInferProperties M3 needs |
| `MemberwiseEmitter.swift` | Text-renders a memberwise strategy to Swift expression text; emitting Swift source is the consuming tools' job, not the strategist contract |

These can be promoted incrementally if a future SwiftInferProperties milestone discovers a need.

## Why this release exists

SwiftInferProperties PRD §11 mandates: *"Generator inference delegates to SwiftProtocolLaws. Don't reimplement the priority list — call into the shared `DerivationStrategist`."* And SwiftInferProperties PRD §5.8 lists *"`DerivationStrategist` exposed publicly from SwiftProtocolLaws (see §11, §21 OQ #4)"* as a hard prerequisite for M3 (contradiction detection). v1.6.0 closes that prerequisite.

The exposure was deliberately deferred until SwiftInferProperties was ready to consume it — keeping the public surface small and matching realised downstream demand. v1.0–v1.5 shipped with `package`-visibility on these types; v1.6 graduates them to `public` because there is now a concrete consumer.

## Upgrade path

From v1.5.0:

1. **No required source changes** for existing consumers of `ProtocolLawKit` or `ProtoLawMacro`. Both shipped products retain their v1.5 surface.
2. **New consumers** (e.g. SwiftInferProperties M3) gain access via `import ProtoLawCore`. The module ships `public` types only; private and `package` symbols remain hidden.
3. **No version-bump required for `ProtocolLawKit`** — its public API is unchanged.

## What's deliberately not in this release

- **Promoting `KnownProtocol` to `public`.** Out of scope until a downstream consumer demonstrably needs it. SwiftInferProperties M3's contradiction detection (PRD §5.6 Equatable contradictions) reads `inheritedTypes` directly via string match; a `KnownProtocol`-keyed lookup is a v1.7 candidate if M3+ uncovers a use case.
- **Promoting `MemberwiseEmitter` to `public`.** It's a text-renderer used only at macro-expansion / discovery-emit time. Downstream consumers reading a `DerivationStrategy` make their own decisions about emit text.
- **`KnownProtocol` cataloguing pass.** No new stdlib protocols added since v1.5.0; the v1.1+ candidates list is still empty per the v1.5.0 notes.
