# PropertyLawKit v1.5.0

**Released:** 2026-04-29
**Tag:** `v1.5.0`
**Previous release:** [v1.3.0](https://github.com/Joseph-Cursio/SwiftPropertyLaws/releases/tag/v1.3.0)

## TL;DR

Closes out the PRD §4.3 v1.1+ candidates list. Ships the entire numeric / integer / FloatingPoint chain (v1.4 — 9 protocols, 61 laws) plus `StringProtocol` (v1.5 — 1 protocol, 8 laws). The kit now covers **27 protocols** with ~75 own Strict-tier laws and ~300 transitively-inherited law-runs across the check matrix. First time the kit's own laws have run end-to-end against stdlib types via the validation harness.

## Highlights

- **v1.4 numeric / integer / FloatingPoint cluster** (M1–M5) — 9 new protocols, 61 new Strict laws, including the `LawCheckOptions.allowNaN` opt-in flag for IEEE-754 NaN-domain laws.
- **v1.5 StringProtocol** — 8 Strict laws covering String round-trip, count consistency, prefix/suffix invariants, case-folding idempotence, and UTF-8 view invariance.
- **First-ever property-based validation of `attaswift/BigInt`** — surfaced one genuine algebraic finding on `BigUInt`'s bitwise NOT (does not satisfy De Morgan's law because `~` operates word-by-word over the underlying word array rather than against a fixed bit-width).
- **Pass 2 stdlib expansion** — `Int32`, `UInt`, `Double`, `String`, `Substring` now run the kit's own laws end-to-end at `.standard` budget (1,000 trials per law).
- **The PRD §4.3 v1.1+ candidates list is empty.** Every protocol the original PRD identified as in-scope has shipped.

## What's new

### v1.5 (this release's headline)

`StringProtocol` (8 own Strict laws):

| Law | What it asserts |
|---|---|
| `stringInitRoundTrip` | `String(String(s)) == String(s)` |
| `countMatchesStringInit` | `s.count == String(s).count` |
| `isEmptyMatchesCountZero` | `s.isEmpty == (s.count == 0)` |
| `hasPrefixEmpty` | `s.hasPrefix("") == true` |
| `hasSuffixEmpty` | `s.hasSuffix("") == true` |
| `lowercasedIdempotent` | `s.lowercased().lowercased() == s.lowercased()` |
| `uppercasedIdempotent` | `s.uppercased().uppercased() == s.uppercased()` |
| `utf8ViewInvariance` | `Array(s.utf8) == Array(String(s).utf8)` |

`.all` chains through to `BidirectionalCollection` (and transitively `Collection` → `Sequence` → `IteratorProtocol`). `Comparable` / `Hashable` / `LosslessStringConvertible` are *not* auto-run — types declaring those alongside `: StringProtocol` still emit independent checks under most-specific dedupe.

### v1.4 (rolled into this release, never separately tagged)

**M1 — algebraic chain (15 laws):**
- `AdditiveArithmetic` — addition associativity / commutativity, zero identity, subtraction inverse, self-subtraction-is-zero
- `Numeric` — multiplication associativity / commutativity, multiplicative identity, zero annihilation, left/right distributivity
- `SignedNumeric` — negation involution, additive inverse, negation distributes over addition, negate-mutation consistency

These laws use exact `==`, so they **do not apply to IEEE-754 floating-point types** — the doc-comments redirect float users to `checkFloatingPointPropertyLaws`, and the macro/discovery side enforces this by having `floatingPoint.subsumedProtocols` include the algebraic chain.

**M2 — integer cluster (19 laws):**
- `BinaryInteger` — 5 division/remainder laws + 9 bitwise (AND/OR/XOR idempotence + commutativity, double-negation, De Morgan, AND-distributes-over-OR, XOR self-is-zero, XOR zero-identity) + shift-by-zero identity + trailing-zero bit-count range
- `SignedInteger` — signum consistency
- `UnsignedInteger` — non-negative + magnitude-is-self

**M3 — FixedWidthInteger (9 laws):**
- bit-width matches type, four `*ReportingOverflow` consistency laws (the `partialValue` half of the tuple matches the wrapping operator), wrapping arithmetic does not trap, min/max bounds reachable, byteSwapped involution, nonzero-bit-count range

**M4 — FloatingPoint (9 always-on + 5 NaN-gated):**
- Always-on: infinity is infinite, signed-zero equality (IEEE-754 mandates `+0 == -0`), additive inverse on finite, next-up/next-down round-trip, sign matches less-than-zero, absolute value non-negative
- `LawCheckOptions.allowNaN`-gated: NaN is NaN, NaN inequality, NaN propagates addition / multiplication, NaN comparison is unordered

**M5 — BinaryFloatingPoint (4 laws):**
- radix-2 constraint, significand/exponent reconstruction, binade membership (`|x.binade| ≤ |x| < 2·|x.binade|` for normal floats — `binade.nextUp` returns the next *representable value* not the next binade), integer-conversion exactness

### Generator helpers

```swift
// Magnitude-bounded integer generators — pick 2^(bitWidth/4) per type so
// triple multiplication doesn't overflow under random sampling.
Gen<Int>.boundedForArithmetic()       // -65_536 ... 65_536
Gen<Int32>.boundedForArithmetic()     // -256 ... 256
Gen<UInt>.boundedForArithmetic()      // 0 ... 65_536

// NaN-injecting FP generators — produce Self.nan on roughly 1 of every 20
// trials so always-on FloatingPoint laws exercise their NaN-skip guards
// even when allowNaN is left at the default `false`.
Gen<Double>.doubleWithNaN()
Gen<Float>.floatWithNaN()
```

## API changes

### New on `LawCheckOptions`

```swift
public var allowNaN: Bool   // default false
```

Default `false` matches v1.3 behavior (no NaN-domain laws run). Set `true` when explicitly testing IEEE-754 NaN behavior on `FloatingPoint` / `BinaryFloatingPoint`.

### New `KnownProtocol` cases

The shared macro/plugin enum gained nine cases for v1.4 plus one for v1.5:

```
.additiveArithmetic, .numeric, .signedNumeric,
.binaryInteger, .signedInteger, .unsignedInteger, .fixedWidthInteger,
.floatingPoint, .binaryFloatingPoint, .stringProtocol
```

Discovery plugin's `swift package propertylawcheck discover` now recognizes inheritance clauses spelling any of these protocols. **Empirical note:** mainstream Swift code rarely declares them — they constrain stdlib generic algorithms, not application types — so a Pass 1 rescan against the originally-surveyed packages (`swift-argument-parser` / `Hummingbird` / `swift-aws-lambda-runtime` / `Sitrep`) produced byte-identical output. The cluster's value lands in the Pass 2 stdlib coverage and external-library validation, not in app-code adoption surfaces.

### No breaking changes

All v1.3.0 call sites continue to compile and pass. The new `allowNaN` field has a default value; new `KnownProtocol` cases are additive; the new check functions add to the public surface without touching existing ones.

## Validation

The Pass 2 sub-package (`Validation/Tests/`) now runs **23 tests across 7 suites** in ≈ 250 ms. New coverage:

- **Stdlib types** (6 tests in `StdlibNumericLawsTests.swift` + 2 tests in `StdlibStringPropertyLawsTests.swift`): `Int32`, `UInt`, `Double` (with and without `allowNaN`), `String`, `Substring`. All pass at `.standard` budget. First time the kit's own laws have run end-to-end against stdlib types via the validation harness.
- **`attaswift/BigInt`** (4 tests): `BigInt` passes cleanly. `BigUInt` requires two `intentionalViolation` suppressions for the bitwise-NOT laws — see the next section.
- **`Foundation.Decimal`** (3 tests): clean — no suppressions needed.
- **`apple/swift-numerics` `Complex<Double>`** (2 tests): passes with 5 FP-rounding suppressions on the algebraic laws (documented expected boundary; same root cause as why `: FloatingPoint` types don't auto-run the algebraic chain).

## Notable finding: `BigUInt` does not satisfy De Morgan's law

Running the kit's `bitwiseDeMorgan` law against `BigUInt` from `attaswift/BigInt` fires a Strict-tier violation:

```
x = 406803, y = 51776
x & y           = 0
~(x & y)        = 0
~x | ~y         = 18446744073709551615   (= UInt64.max)
```

Root cause: `BigUInt` stores its value as `[UInt64]` and `~` operates word-by-word on that storage. The storage size depends on the value, so the two sides of De Morgan's identity end up flipping different numbers of bits. `BigInt` (signed) is unaffected because `~x = -(x+1)` is a value-level arithmetic identity that doesn't depend on storage.

Full write-up in `Validation/findings/BigUInt-bitwise-divergence.md`. The Pass 2 test ships with the `intentionalViolation` suppressions documented inline so the suite still passes; the suppressions are an explicit policy decision visible in the suppression record, not a silent skip.

This is the kit's **first algebraic finding on a real-world external package** — likely the first property-based test of `attaswift/BigInt`'s protocol conformance ever published.

## Coverage

```
Region:    84.47%    (1906 regions, 296 missed)
Function:  88.37%    (843 functions, 98 missed)
Line:      90.29%    (6569 lines, 638 missed)
```

The bulk of un-hit code is in `formatCounterexample` closures that only execute on violations — stdlib types pass all the laws, so these are dead in the unit-test path. Planted-bug fixtures cover the formatters for v1.0–v1.3 protocols; v1.4 / v1.5 protocols use stdlib types as the reference implementations because custom conformers (custom FixedWidthInteger / FloatingPoint / StringProtocol) are essentially nonexistent in the Swift ecosystem and full conformance boilerplate is ~150–250 lines per violator.

## What's deliberately not in this release

- **`Sendable` law check** — value semantics is a heuristic, not a checkable invariant from outside the type. A `checkSendableContractLaws` Heuristic-tier suite (mutation-after-share spot checks) is named in PRD §4.3 Coverage Scope as a research item, not a deliverable.
- **`LazySequenceProtocol` / `LazyCollectionProtocol`** — laziness is a property to test (no eager evaluation) but requires instrumentation hooks the kit doesn't currently have.
- **`CustomStringConvertible` / `CustomDebugStringConvertible` / `TextOutputStreamable` / `TextOutputStream`** — output formatting protocols with no semantic laws beyond "the conformance doesn't crash."
- **`SwiftInferProperties`** — the downstream package (PRD Contribution 2) remains in proposal-only state. Separate, larger lift.

## Upgrade path

From v1.3.0:

1. No required source changes. Existing call sites continue to compile.
2. If you have custom `KnownProtocol`-shaped switches (most users don't), add cases for the new entries.
3. Optional: adopt the new generator helpers (`boundedForArithmetic()`, `doubleWithNaN()` / `floatWithNaN()`) in your own tests when checking the algebraic / FloatingPoint laws.

## Acknowledgments

The v1.4 cluster's design — particularly the call to NOT auto-run the algebraic chain from `FloatingPoint` — was sketched by the Plan agent against PRD §4.3 v1.1+ candidates. The rationale held up empirically: the BigInt validation finding (where De Morgan fails on `BigUInt`'s storage-dependent `~`) and the Complex<Double> rounding suppressions both confirm the design's predicted boundaries.
