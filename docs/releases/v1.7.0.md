# SwiftProtocolLaws v1.7.0

**Released:** 2026-05-01
**Tag:** `v1.7.0`
**Previous release:** [v1.6.0](https://github.com/Joseph-Cursio/SwiftProtocolLaws/releases/tag/v1.6.0)

## TL;DR

Kit-prep release for SwiftInferProperties M5. Promotes `MemberwiseEmitter` from `package` to `public` and extracts a unified `GeneratorExpressionEmitter` into `ProtoLawCore` so SwiftInferProperties' lifted-test stub writeout can emit byte-identical generator code to the macro and the discovery plugin without duplicating the strategy → Swift-source mapping. Adds a downstream-consumer smoke test (no `@testable`) pinning the public `PropertyBackend` + `Seed` + `BackendCheckResult` surface against accidental tightening. No behaviour change visible to existing consumers of `ProtocolLawKit` or `ProtoLawMacro`.

## What's new

### Promoted to `public` in `ProtoLawCore`

| Symbol | Source file | Purpose |
|---|---|---|
| `MemberwiseEmitter.expression(typeName:members:)` | `MemberwiseEmitter.swift` | `zip(Gen<…>(), …).map { Type(member: $0.N, …) }` text for memberwise-Arbitrary strategies. Was `package`-scoped; documented in the v1.6.0 notes as a v1.7 candidate. |
| `GeneratorExpressionEmitter.expression(typeName:strategy:)` (new) | `GeneratorExpressionEmitter.swift` (new) | Single source of truth for `DerivationStrategy → Swift expression text`. Covers all five strategy arms. |

### Internal consumers consolidated

Both `ProtoLawMacroImpl.ProtoLawSuiteMacro` and `ProtoLawDiscoveryTool.GeneratedFileEmitter` previously carried their own copy of the strategy → Swift-source switch. Both now call `GeneratorExpressionEmitter.expression`. As a side effect, an existing inter-consumer drift collapses: the macro emitted `.compactMap` on a fresh line for `.rawRepresentable` (line-width concern documented in its docstring); the discovery plugin emitted it on the same line. Both now use the macro's multi-line shape — verified against existing `EmitterGoldenTests` (`.contains`-style, no byte-stable equality, so the layout shift is invisible) and the macro's `MacroExpansionMultiTypeTests` byte-stable goldens (already pinned to the multi-line form).

### Downstream-consumer smoke test

New `Tests/ProtocolLawKitTests/DownstreamPropertyBackendSmokeTests.swift` exercises the public surface SwiftInferProperties M5 will use:

- `Seed(stateA:stateB:stateC:stateD:)`
- `SwiftPropertyBasedBackend()` + the `init(identifier:)` overload
- `backend.check(...)` → `.passed(trialsRun:initialSeed:)` on always-true property
- `backend.check(...)` → `.failed(trialsRun: 1, …, thrownError: nil)` on always-false property (short-circuit verification)
- `backend.check(...)` → `.failed(…, thrownError: ErrorBox(…))` on throwing property; `ErrorBox.message` preserves the error type's textual description
- Replay reproducibility — same `Seed` → same first failing input across two independent `check` calls

Uses plain `import ProtocolLawKit` + `import PropertyBased` (not `@testable`) so anything accidentally not-public surfaces here as a compile error rather than waiting until SwiftInferProperties M5 tries to consume it. Mirrors the M3.1 `DerivationStrategistSmokeTests` pattern in the downstream package.

### Stayed `package`-scoped

| File | Why |
|---|---|
| `KnownProtocol.swift` | Internal stdlib-protocol lookup table consumed only by `ProtoLawMacroImpl` and `ProtoLawDiscoveryTool`. SwiftInferProperties' contradiction detection (M3) reads inheritance via string match; no concrete downstream need has surfaced. v1.8+ candidate if M5–M7 needs it. |

## Why this release exists

SwiftInferProperties M5 (PRD §5.7 + §5.8) writes lifted property-test stubs to `Tests/Generated/SwiftInfer/`. Each stub references a generator expression sized to the property under test — and the canonical mapping from a `DerivationStrategy` to that expression text already lives in this repo (the macro and the discovery plugin have been emitting it since v1.3's memberwise-Arbitrary cluster). Without v1.7.0, SwiftInferProperties M5 would have to duplicate the strategy → Swift-source mapping, accruing drift the same way the macro and discovery plugin themselves did before this release.

The exposure was deliberately deferred until SwiftInferProperties was ready to consume it — keeping the public surface small and matching realised downstream demand. The v1.6.0 release notes explicitly listed `MemberwiseEmitter` promotion as a v1.7 candidate "if a future SwiftInferProperties milestone discovers a need." M5 is that milestone.

## Upgrade path

From v1.6.0:

1. **No required source changes** for existing consumers of `ProtocolLawKit` or `ProtoLawMacro`. The macro's emitted output is byte-identical across all five strategy arms. The discovery plugin's emitted output for `.rawRepresentable` strategies shifts from a single-line `.compactMap` lift to a multi-line one — the same Swift code, different layout, no behaviour change.
2. **New downstream consumers** can call `GeneratorExpressionEmitter.expression(typeName:strategy:)` for the full mapping or `MemberwiseEmitter.expression(typeName:members:)` for the memberwise-only sub-mapping, both via `import ProtoLawCore`.
3. **No version-bump required for `ProtocolLawKit` or `ProtoLawMacro`** — their public APIs are unchanged.

## What's deliberately not in this release

- **`Seed(uint64Compact:)` convenience init.** SwiftInferProperties' M4.3 sampling seed is a single `UInt64`; `Seed` carries the four `UInt64`s of a Xoshiro256\*\* state. Two paths exist for M5: widen the SwiftInfer-side `SamplingSeed.derive` to 256 bits (pull more bytes from the SHA256 digest), or splat the `UInt64` into `stateA` and zero the other three. The K-prep-M2 smoke test verified the splat path works (`Seed(stateA: 0xABCDEF1234567890, stateB: 0, stateC: 0, stateD: 0)`); whether to widen on the consumer side or add a kit-side convenience is an M5-side decision. v1.8 candidate if the kit-side convenience wins.
- **`KnownProtocol` promotion.** See the table above.
- **CI-side automation around `swift test` for the downstream-consumer smoke.** The smoke runs as part of the kit's existing `swift test` invocation; a separate "import-as-package-dep and re-run" job is overkill until SwiftInferProperties is on a published kit version rather than a local-path dep.
