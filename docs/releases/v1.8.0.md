# SwiftProtocolLaws v1.8.0

**Released:** 2026-05-02
**Tag:** `v1.8.0`
**Previous release:** [v1.7.0](https://github.com/Joseph-Cursio/SwiftProtocolLaws/releases/tag/v1.7.0)

## TL;DR

First **kit-defined** protocol cluster — every prior release covered stdlib-defined protocols only. Adds `Semigroup` and `Monoid` to `ProtocolLawKit` so SwiftInferProperties' RefactorBridge (M7) can write conformances the kit verifies on every CI run, closing the discovery → conformance → law-check loop for merge-shape types that `AdditiveArithmetic` / `SetAlgebra` / `Numeric` don't fit. Two new protocols, three new Strict-tier laws, zero changes required in `ProtoLawSuiteMacro` or `ProtoLawDiscoveryTool`.

## What's new

### `Semigroup` (kit-defined, in `ProtocolLawKit`)

```swift
public protocol Semigroup {
    static func combine(_ lhs: Self, _ rhs: Self) -> Self
}
```

One Strict-tier law:

- `Semigroup.combineAssociativity` — `combine(combine(a, b), c) == combine(a, combine(b, c))`

Verified by `checkSemigroupProtocolLaws<Value: Semigroup & Equatable & Sendable, Shrinker>(...)`. `Equatable` is required to *check* the law but not to *declare* the conformance — the protocol decl deliberately does not refine `Equatable`.

### `Monoid` (kit-defined, in `ProtocolLawKit`)

```swift
public protocol Monoid: Semigroup {
    static var identity: Self { get }
}
```

Two Strict-tier own laws plus the inherited Semigroup chain via `.all`:

- `Monoid.combineLeftIdentity` — `combine(.identity, a) == a`
- `Monoid.combineRightIdentity` — `combine(a, .identity) == a`

Verified by `checkMonoidProtocolLaws(..., laws: LawSelection = .all)`. `.ownOnly` skips the inherited Semigroup check, mirroring the convention from `Numeric` / `BinaryFloatingPoint` / `StringProtocol`.

### Naming choices

- **`combine` rather than an operator (`+`, `<>`):** doesn't collide with arithmetic `+`; most semigroup-shaped Swift types use names like `merge`, `concat`, `union` and don't fit additive semantics.
- **`identity` rather than `empty` / `zero`:** `empty` overlaps with `RangeReplaceableCollection.init()`; `zero` overlaps with `AdditiveArithmetic.zero`. `identity` is unambiguous in the algebraic-structure context.

When SwiftInferProperties' RefactorBridge (M7.5) writes a conformance for a type whose existing identity has a different name, it bridges via a one-line static aliasing — the kit protocol stays rigid, naming flexibility lives downstream.

### `KnownProtocol` — adds `.semigroup` and `.monoid`

`ProtoLawCore.KnownProtocol` gains two cases plus their `typeNameMap`, `subsumedProtocols`, `checkFunctionName`, `declarationName`, and `testNameFragment` switch arms. `subsumedProtocols` maps `.monoid → [.semigroup]` so a type spelled `: Semigroup, Monoid` collapses under most-specific dedupe to a single `checkMonoidProtocolLaws` call.

`ProtoLawSuiteMacro` and `ProtoLawDiscoveryTool.GeneratedFileEmitter` required **zero source changes** — both consume `KnownProtocol` generically and pick the new cases up automatically. v1.8.3's integration tests confirm this end-to-end: macro expansion of `: Semigroup, Monoid` emits one `checkMonoidProtocolLaws` call; the discovery scanner walking real source files reduces the inheritance clause to `[.monoid]` post-`mostSpecific`.

### Planted-bug coverage

Per PRD §8 framework self-test gate:

- `NonAssociativeCombine` — combine floor-divides by 2 when both operands are non-zero (mirroring `NonAssociativeAddition`'s trick from v1.4 M1). Confirms `combineAssociativity` fires.
- `BadLeftIdentity` / `BadRightIdentity` — bump-by-1 on the relevant identity case while preserving the other side. Confirms each Monoid own law fires independently.

## Why this release exists

PRD §11 (and its mirror in `docs/SwiftInferProperties.md`) describes the bidirectional loop: SwiftInfer detects algebraic structure → suggests protocol conformance → SwiftProtocolLaws' discovery plugin emits the law-check on the next regeneration → laws are enforced on every CI run thereafter.

Until v1.8, that loop closed for **stdlib-mappable** algebraic structures (Monoid → `AdditiveArithmetic` if `+`/`zero`-shaped, BoundedJoinSemilattice → `SetAlgebra`, Ring → `Numeric`) but not for the more common Swift case: types with `merge` / `combine` / `concat` operations that don't fit `+`. Those claims stayed informational ("looks like a Semigroup" as English text) with no actual conformance for the kit to verify. v1.8 fixes that.

This is also the first time the kit defines its own protocols rather than only verifying stdlib ones. The same architectural invariant holds: adding a protocol = one `KnownProtocol` case + four switch arms + a Public/<Protocol>Laws.swift file. v1.8 validated that pattern works for kit-defined protocols too.

## Upgrade path

From v1.7.0:

1. **No source changes for existing consumers.** `Semigroup` and `Monoid` are additive — the kit's existing protocol checks are unchanged.
2. **New downstream consumers:**
   ```swift
   import ProtocolLawKit

   struct Counter: Monoid, Equatable {
       var count: Int
       static let identity = Counter(count: 0)
       static func combine(_ lhs: Counter, _ rhs: Counter) -> Counter {
           Counter(count: lhs.count + rhs.count)
       }
   }

   try await checkMonoidProtocolLaws(
       for: Counter.self,
       using: Counter.gen()
   )
   ```
3. **`@ProtoLawSuite`-marked types** with `: Semigroup` or `: Monoid` in their inheritance clause now generate the corresponding `check…` calls automatically.
4. **`swift package protolawcheck discover`** picks up `: Semigroup` / `: Monoid` types in the scanned target the same way it picks up stdlib protocols.

## What's deliberately not in this release

- **`Group`, `Semilattice`, `Ring`.** Wider design work — Group needs an `inverse` requirement; Semilattice adds idempotence; Ring is two interacting structures (additive abelian group + multiplicative semigroup with distributivity). Claim-only stays for these. Land when SwiftInferProperties M8's algebraic-structure-composition pass motivates them. Probably a v1.10 cluster.
- **`CommutativeSemigroup` / `CommutativeMonoid`.** Independent axis from associativity / identity. SwiftInferProperties' commutativity template fires with a `+30` weight today; promoting to a kit conformance is an M8 question.
- **Operator sugar (`infix operator <>`, `+`-overload on Semigroup).** Stuck with `static func combine(_:_:)`. Operators add naming-conflict risk and aren't needed for law verification. Reconsider only if real-user adoption flags ergonomics in a v1.9+ cluster.
- **`Functor`, `Applicative`, `Monad`.** Out of scope indefinitely unless a strong use case emerges.
- **`DerivationStrategist` arms for Semigroup/Monoid types.** Conforming to Semigroup doesn't help generate values of the type — it composes existing ones. The `.derived` priority list (PRD §5.7) stays as-is.
- **Kit's `--advisory` mode suggesting `: Semigroup`.** Overlaps SwiftInferProperties' TemplateEngine — kept on the SwiftInfer side.
- **Validation-pass coverage of the new laws.** Pass 1/2/3 currently exercise stdlib protocols. Real-world Swift packages with `: Semigroup`-shaped types don't exist yet — the protocol just shipped. Defer to post-1.8 adoption metrics; `Validation/FINDINGS.md` gets a new entry once a downstream user adopts.
- **Performance budget re-validation.** Semigroup associativity is 3-sample / 1-property; Monoid identity laws are 1-sample / 1-property each. Same shape as `AdditiveArithmetic` — no new perf characterization needed.
- **Migration tool for libraries already shipping their own `Semigroup` protocol.** Some Swift functional-programming libraries do; no automated migration path. The kit's `Semigroup` is the canonical SwiftProtocolLaws-recognized form going forward.
