# PropertyLawKit v1.8 ŌĆö Kit-Defined Algebraic Protocols (Semigroup + Monoid)

Drafted 2026-05-02 against repo state at commit `HEAD` (post-v1.7).

## Why this cluster

SwiftInferProperties M7's RefactorBridge needs real protocol conformances to write ŌĆö the `LiftedConformanceEmitter` (M7.4, already shipped at SwiftInferProperties commit `67bfe6b`) emits `extension TypeName: Semigroup {}` / `Monoid {}`. Per **SwiftInferProperties PRD v0.4 ┬¦11**, the design intent is "SwiftInfer detects algebraic structure ŌåÆ suggests protocol conformance ŌåÆ SwiftPropertyLaws' discovery plugin emits the law-check on the next regeneration ŌåÆ laws are enforced on every CI run thereafter." For Semigroup / Monoid claims the kit currently has no protocol to verify against, breaking that loop.

**SwiftInferProperties PRD v0.2 ┬¦5.4** (referenced as "unchanged" in v0.4 ┬¦5.4) lists Semigroup as "(no stdlib protocol ŌĆö claim only)" and Monoid as "suggest `AdditiveArithmetic` if `+`/`zero`-shaped, otherwise informational." That posture leaves merge-shape types (the common Swift case ŌĆö `merge`, `combine`, `concat`, `union`) with no conformance path. v1.8 closes that gap by adding the protocols to the kit.

This is the kit's first **kit-defined** protocol cluster ŌĆö every prior cluster (v1.0ŌĆōv1.7) covered stdlib-defined protocols only. v1.8 establishes the precedent that the kit can add protocols where the stdlib leaves a useful gap.

## Scope

Two protocols, two own + one inherited Strict laws total:

| Protocol | Requirement | Own laws | Inherited |
|---|---|---|---|
| `Semigroup` | `static func combine(_ lhs: Self, _ rhs: Self) -> Self` | `combineAssociativity` | (none) |
| `Monoid: Semigroup` | adds `static var identity: Self { get }` | `combineLeftIdentity`, `combineRightIdentity` | Semigroup chain via `subsumedProtocols` |

`combineAssociativity`: `combine(combine(a, b), c) == combine(a, combine(b, c))` ŌĆö three-sample, requires `Equatable`.
`combineLeftIdentity`: `combine(.identity, a) == a` ŌĆö one-sample.
`combineRightIdentity`: `combine(a, .identity) == a` ŌĆö one-sample.

## Design decisions

### Why `combine(_:_:)` instead of an operator

- Doesn't collide with `+` (some semigroups aren't additive ŌĆö string concat, set union, max).
- Matches SwiftIdempotency's `@Idempotent` convention and idiomatic Swift static-method style.
- Operators (`<>`, `++`) can land later if real-user adoption flags ergonomics.

### Why `identity` (not `empty` / `zero`)

- `empty` overlaps with `RangeReplaceableCollection.init()` and `Set().isEmpty` semantics.
- `zero` overlaps with `AdditiveArithmetic.zero` ŌĆö and most monoids aren't additive.
- `identity` is unambiguous in the algebraic-structure context.
- Naming flexibility lives **on the SwiftInferProperties side**: the M7.5 RefactorBridge accept path will write a bridging extension when the user's existing identity has a different name:
  ```swift
  extension Counter: Monoid {
      public static var identity: Counter { .empty }   // user's existing static
      public static func combine(_ lhs: Counter, _ rhs: Counter) -> Counter {
          merge(lhs, rhs)                              // user's existing op
      }
  }
  ```
  The kit protocol stays rigid; SwiftInferProperties knows the user's curated `.empty` / `.zero` / `.identity` / `.none` / `.default` verbs (from `IdentityElementTemplate`) and aliases them at writeout time.

### Why no `CommutativeSemigroup` / `CommutativeMonoid`

Commutativity is structurally orthogonal to associativity / identity. SwiftInfer's commutativity template fires with a `+30` weight today; whether to upgrade those into kit conformances is an M8 (algebraic-structure-composition) question. v1.8 stays minimal.

## What this cluster will NOT do

### Out of scope, but reachable in a future cluster

1. **`Group`, `Semilattice`, `Ring`.** Wider design: Group needs an `inverse` requirement; Semilattice adds idempotence; Ring is two interacting structures (additive abelian group + multiplicative semigroup with distributivity). Claim-only stays for these. Land when SwiftInferProperties M8 motivates them. Probably a v1.10 cluster.

2. **`CommutativeSemigroup` / `CommutativeMonoid`.** Independent axis from associativity / identity. SwiftInferProperties' commutativity template fires with `+30` today; promoting to a kit conformance is an M8 question.

3. **Operator sugar (`infix operator <>`, `+`-overload on Semigroup).** Stick with `static func combine(_:_:)`. Adds naming-conflict risk and isn't needed for law verification. Reconsider only if real-user adoption flags ergonomics in a v1.9+ cluster.

### Permanently out of scope (kit-side)

4. **`Functor`, `Applicative`, `Monad`.** Even more abstract; no clear demand from real Swift code today. Out of scope indefinitely unless a strong use case emerges. Cross-reference to `Swift Standard Library Protocols.md` would note "intentionally not covered."

5. **Cross-protocol law checks** (e.g., "this Monoid is also a SetAlgebra ŌĆö verify the two algebraic structures are compatible"). The kit's existing `mostSpecific` deduplication handles per-protocol emit, but cross-structure compatibility laws are a new genre. Defer to whenever Ring/Semilattice land ŌĆö they need it more.

6. **`DerivationStrategist` arms for Semigroup/Monoid types.** Conforming to Semigroup doesn't help generate values of the type ŌĆö it composes existing ones. The `.derived` priority list (PRD ┬¦5.7) stays as-is.

7. **Kit's `--advisory` mode suggesting `: Semigroup`.** The M4 Advisory layer suggests stdlib conformances syntactically. Adding "this looks like a Semigroup" overlaps SwiftInferProperties' TemplateEngine ŌĆö keep that responsibility on the SwiftInfer side.

### Out of scope for v1.8 specifically; reconsider post-release

8. **Validation-pass coverage of the new laws.** Pass 1/2/3 currently exercise stdlib protocols. Finding real-world Swift packages with `: Semigroup`-shaped types pre-release is impossible since the protocol doesn't exist anywhere yet. Defer to post-1.8 adoption metrics ŌĆö `Validation/FINDINGS.md` gets a new entry once a downstream user adopts it (likely SwiftInferProperties' own validation corpus once M7.6 lands).

9. **Performance budget re-validation.** Semigroup associativity is 3-sample / 1-property; Monoid identity laws are 1-sample / 1-property each. Same shape as `AdditiveArithmetic` ŌĆö no new perf characterization needed under the default `.standard` 1000-trial budget.

10. **Migration tool for libraries already using ad-hoc `Semigroup` protocols.** Some Swift functional-programming libraries ship their own `Semigroup`. No automated migration path. Documentation note in v1.8.0 release notes pointing at the kit's protocol as the canonical SwiftPropertyLaws-recognized form.
