# ProtocolLawKit v1.4 — Numeric/Arithmetic Cluster Plan

Drafted 2026-04-29 by Plan agent against repo state at commit `2a28c64`.

Cadence matches v1.1 (one protocol per milestone, three sub-commits each) and v1.2 (one cluster commit). Recommend the v1.1 shape — the nine protocols span four chains with very different shrink/generator concerns; batching them all into one v1.2-style commit would lose review-ability.

## 1. Protocol inheritance map + subsumption ordering

Stdlib reality (single source of truth, not aspirational):

```
AdditiveArithmetic                   (root of the algebraic chain)
└── Numeric                          (refines AdditiveArithmetic + ExpressibleByIntegerLiteral)
    ├── SignedNumeric                (adds prefix `-`)
    │   ├── SignedInteger            (refines BinaryInteger + SignedNumeric)
    │   └── FloatingPoint            (refines SignedNumeric + Strideable + Hashable)
    │       └── BinaryFloatingPoint  (refines FloatingPoint + ExpressibleByFloatLiteral)
    └── BinaryInteger                (refines Numeric + Hashable + Strideable + CustomStringConvertible)
        ├── SignedInteger            (also under SignedNumeric — diamond)
        ├── UnsignedInteger
        └── FixedWidthInteger        (refines BinaryInteger; Int/UInt/Int8…64 conform here, plus their `&System` siblings)
```

Two diamonds matter: `SignedInteger : BinaryInteger & SignedNumeric` and `FixedWidthInteger` is orthogonal to `Signed`/`Unsigned` (a type can be both `FixedWidthInteger & SignedInteger` like `Int32`, or `FixedWidthInteger & UnsignedInteger` like `UInt`).

Subsumption entries to add to `KnownProtocol.subsumedProtocols` at `Sources/ProtoLawCore/KnownProtocol.swift:131`:

| Case | subsumedProtocols |
|---|---|
| `.additiveArithmetic` | `[]` |
| `.numeric` | `[.additiveArithmetic]` |
| `.signedNumeric` | `[.numeric, .additiveArithmetic]` |
| `.binaryInteger` | `[.numeric, .additiveArithmetic]` (plus implies `.strideable`/`.hashable` — see `set(from:)` note below) |
| `.signedInteger` | `[.binaryInteger, .signedNumeric, .numeric, .additiveArithmetic]` |
| `.unsignedInteger` | `[.binaryInteger, .numeric, .additiveArithmetic]` |
| `.fixedWidthInteger` | `[.binaryInteger, .numeric, .additiveArithmetic]` (Signed/Unsigned are siblings — like Mutable/RangeReplaceable in v1.2) |
| `.floatingPoint` | `[.signedNumeric, .numeric, .additiveArithmetic]` |
| `.binaryFloatingPoint` | `[.floatingPoint, .signedNumeric, .numeric, .additiveArithmetic]` |

`unemittable`: empty additions. None of these check signatures need extra associated-type generators (unlike Strideable's `strideGenerator`). However — `BinaryInteger` and `FloatingPoint` already inherit `Strideable` in stdlib, which is on the unemittable list. The plugin must not auto-emit a `Strideable` call from `: BinaryInteger`. Two choices:

- **(A)** Extend `KnownProtocol.set(from:)` at `KnownProtocol.swift:62` so that seeing `.binaryInteger` / `.floatingPoint` injects `.strideable` then immediately suppresses it via `unemittable` — same shape as the existing Strideable→Comparable line at `KnownProtocol.swift:76-78`. Auto-injecting `.hashable` from `.binaryInteger` is reasonable too (BinaryInteger refines Hashable in stdlib).
- **(B)** Don't inject anything; the type's own `: Hashable` clause (if present) does the work.

Recommend (A) for `.hashable` only — it's emit-able and useful — and skip `.strideable` injection because Strideable is unemittable anyway. This matches the v1.1 M1 "auto-add Comparable when Strideable is seen" rationale.

A type written as `: FixedWidthInteger & SignedInteger` should after `mostSpecific` collapse to `{.fixedWidthInteger, .signedInteger}` (siblings), with everything below removed. Verify against the v1.2 sibling pattern (Mutable + Bidirectional emit both checks per `KnownProtocol.swift:141-142`).

`from(typeName:)` map at `KnownProtocol.swift:35-53` adds nine string entries. `checkFunctionName`, `declarationName`, `testNameFragment` — three switch arms each. Matches the v1.1 shape exactly.

## 2. Law inventory — Strict tier only

Keep the file naming flat in `Sources/ProtocolLawKit/Public/`, one file per protocol matching `EquatableLaws.swift` / `SetAlgebraLaws.swift`.

### `AdditiveArithmeticLaws.swift`
- `additionAssociativity` — `(x + y) + z == x + (y + z)`
- `additionCommutativity` — `x + y == y + x`
- `zeroAdditiveIdentity` — `x + .zero == x`
- `subtractionInverse` — `(x + y) - y == x`
- `selfSubtractionIsZero` — `x - x == .zero`

### `NumericLaws.swift` (extends AdditiveArithmetic)
- `multiplicationAssociativity` — `(x * y) * z == x * (y * z)` (note: under fixed-width signed types this can overflow → trap. Keep generators bounded; document that callers using `Int` should pass a clamped generator like `Gen<Int>.int(in: -1000...1000)` for `.standard` budget. See §4 below.)
- `multiplicationCommutativity` — `x * y == y * x`
- `oneMultiplicativeIdentity` — `x * 1 == x` (`1` via `ExpressibleByIntegerLiteral` requirement)
- `zeroAnnihilation` — `x * 0 == 0`
- `leftDistributivity` — `x * (y + z) == x * y + x * z`
- `rightDistributivity` — `(x + y) * z == x * z + y * z`

### `SignedNumericLaws.swift` (extends Numeric)
- `negationInvolution` — `-(-x) == x`
- `additiveInverse` — `x + (-x) == .zero`
- `negationDistributesOverAddition` — `-(x + y) == (-x) + (-y)`
- `negateMutationConsistency` — `var y = x; y.negate(); y == -x`

### `BinaryIntegerLaws.swift` (extends Numeric)

Five clusters; lots of laws but each is one liner.

Division/remainder:
- `divisionMultiplicationRoundTrip` — for `y != 0`: `(x / y) * y + (x % y) == x`
- `remainderRangeStrict` — `y != 0` ⇒ `abs(x % y) < abs(y)`
- `selfDivisionIsOne` — `x != 0` ⇒ `x / x == 1`
- `divisionByOneIdentity` — `x / 1 == x`
- `quotientAndRemainderConsistency` — `x.quotientAndRemainder(dividingBy: y) == (x/y, x%y)`

Bitwise:
- `bitwiseAndIdempotence` — `x & x == x`
- `bitwiseOrIdempotence` — `x | x == x`
- `bitwiseAndCommutativity` — `x & y == y & x`
- `bitwiseOrCommutativity` — `x | y == y | x`
- `bitwiseXorSelfIsZero` — `x ^ x == 0`
- `bitwiseXorZeroIdentity` — `x ^ 0 == x`
- `bitwiseDoubleNegation` — `~~x == x`
- `bitwiseAndDistributesOverOr` — `x & (y | z) == (x & y) | (x & z)`
- `bitwiseDeMorgan` — `~(x & y) == ~x | ~y`

Shift:
- `leftShiftThenRightShiftRoundTrip` — for `n` in `0..<bitWidth-someMargin` with no overflow into sign: `(x << n) >> n == x` (bound `x`'s magnitude per shift count). Convention: only test for unsigned to avoid sign-extension trickery, or gate on positive `x`. Document as Strict-with-bounded-generator.
- `shiftByZeroIdentity` — `x << 0 == x` and `x >> 0 == x`

Bit-counting:
- `trailingZeroBitCountRange` — `0 <= x.trailingZeroBitCount && x.trailingZeroBitCount <= x.bitWidth`
- `nonzeroBitCountRange` — `0 <= x.nonzeroBitCount && x.nonzeroBitCount <= x.bitWidth`

### `SignedIntegerLaws.swift` (extends BinaryInteger + SignedNumeric)
Most signed-integer laws are inherited from SignedNumeric + BinaryInteger. SignedInteger's own contract is mostly representational — leave own-table at:
- `signednessConsistency` — `(x < 0) == x.signum() == -1` and `(x > 0) == x.signum() == 1` and `(x == 0) == x.signum() == 0`

A thin own-table is fine — see `MutableCollectionLaws` (only 2 own laws) for precedent.

### `UnsignedIntegerLaws.swift` (extends BinaryInteger)
- `nonNegative` — `x >= 0` (trivially holds; serves as a self-test gate against custom UnsignedInteger conformers that lie about signedness)
- `magnitudeIsSelf` — `x.magnitude == x` (UnsignedInteger has `Magnitude == Self`)

### `FixedWidthIntegerLaws.swift` (extends BinaryInteger)
- `bitWidthMatchesType` — `x.bitWidth == Self.bitWidth` (instance bitWidth is constant for fixed-width)
- `addingReportingOverflowConsistency` — when `(x &+ y)` doesn't overflow, `x.addingReportingOverflow(y) == (x + y, false)`. When it overflows, the wrapped value matches `x &+ y`.
- `subtractingReportingOverflowConsistency` — symmetric.
- `multipliedReportingOverflowConsistency` — symmetric.
- `dividedReportingOverflowOnDivByZero` — `x.dividedReportingOverflow(by: 0).overflow == true`.
- `wrappingArithmeticDoesNotTrap` — `x &+ y`, `x &- y`, `x &* y` always produce a value (no-throw assertion; the generator just has to feed extreme values).
- `minMaxBoundsAreReachable` — every generated `x` satisfies `Self.min <= x && x <= Self.max`.
- `byteSwappedInvolution` — `x.byteSwapped.byteSwapped == x`.

### `FloatingPointLaws.swift` (extends SignedNumeric)

Always-on (Strict, no NaN gating needed because the law is true even in NaN's presence):
- `infinityIsInfinite` — `Self.infinity.isInfinite`
- `negativeInfinityComparison` — `-Self.infinity < Self.infinity`
- `zeroIsZero` — `(Self.zero).isZero`
- `signedZeroEquality` — `Self.zero == -Self.zero` (IEEE-754 mandates `+0 == -0`)
- `roundedZeroIdentity` — `(0 as Self).rounded() == 0`
- `additiveInverseFinite` — `x.isFinite ⇒ x + (-x) == 0`
- `nextUpDownRoundTrip` — `x.isFinite && x != .greatestFiniteMagnitude ⇒ x.nextUp.nextDown == x`
- `signMatchesIsLessThanZero` — `x.sign` consistent with `isLessThan` logic on finite values

`.allowNaN`-gated (Strict tier; only run when option is set):
- `nanIsNaN` — `Self.nan.isNaN`
- `nanInequality` — `let n = Self.nan; n != n` (canonical IEEE-754 behavior; this fails Equatable.reflexivity by design — that's why this is gated)
- `nanPropagatesAddition` — `(Self.nan + x).isNaN`
- `nanPropagatesMultiplication` — `(Self.nan * x).isNaN`
- `nanComparisonIsUnordered` — `!(Self.nan < x) && !(Self.nan > x) && !(Self.nan == x)`
- `absoluteValueNonNegative` — `x.magnitude >= 0` (NaN's magnitude is NaN — fails for NaN, hence gated)

### `BinaryFloatingPointLaws.swift` (extends FloatingPoint)
- `significandExponentReconstruction` — for finite non-zero `x`: `Self(sign: x.sign, exponent: x.exponent, significand: x.significand) == x`
- `binadeMembership` — `x.isFinite && !x.isZero ⇒ x.binade <= abs(x) && abs(x) < x.binade.nextUp`
- `convertingFromIntegerExactness` — for small integers `n` where `|n| < 2^significandBitCount`: `Self(exactly: n).map(Int.init) == n`
- `radix2Constraint` — `Self.radix == 2` (binade math depends on this)

## 3. Milestone breakdown M1–M5

Each milestone follows the v1.1 three-commit shape (`(1/3) laws`, `(2/3) wire macro+discovery`, `(3/3) docs`).

| M | Protocol(s) | Sequencing |
|---|---|---|
| **M1** | `AdditiveArithmetic`, `Numeric`, `SignedNumeric` | Land as one milestone — they share generator concerns and the inheritance chain is linear. Three protocols × ~3-4 own laws each = ~10 laws — comparable to v1.2's volume. |
| **M2** | `BinaryInteger` + `SignedInteger` + `UnsignedInteger` | Together. UnsignedInteger and SignedInteger own-tables are tiny (1-2 laws); BinaryInteger is the load-bearing one. All three share `BinaryInteger`'s division/bitwise generator concerns. |
| **M3** | `FixedWidthInteger` | Standalone — overflow-reporting laws need different test scaffolding (`addingReportingOverflow` returns a tuple, planted-bug shape is distinct). |
| **M4** | `FloatingPoint` | Standalone — `.allowNaN` gating mechanics (see §5) lands here. This milestone introduces a new field on `LawCheckOptions`. |
| **M5** | `BinaryFloatingPoint` + Pass 2 expansion | Land BinaryFloatingPoint laws and extend `Validation/Tests` to exercise `Int32` (FixedWidthInteger + SignedInteger) and `Double` (BinaryFloatingPoint) directly — mirroring how Pass 2 was extended for RawRepresentable in commit `a936a62`. |

**Sequential vs parallel.** M1 → M2 → M3 must be strictly sequential because `BinaryInteger` law file imports `Numeric` law signatures via inherited-suite dispatch. M4 can land in parallel with M3 in a feature branch — they share no code paths. M5 depends on M4 (`BinaryFloatingPoint` chains through `FloatingPoint`) and on M3 (Pass 2 expansion needs `FixedWidthInteger` to exist).

**Macro/plugin wiring strategy.** Recommend incremental per milestone, not batched. Three reasons:

1. The KnownProtocol diff per milestone is small (1-3 cases, 3-4 lines each switch) — lower review burden than nine cases at once.
2. Per-milestone integration tests (`MacroExpansionTests`, `EmitterGoldenTests`) catch wiring bugs immediately rather than at the end.
3. Match the v1.1 shape — every M1-M5 in v1.1 had its own `(2/3) wire into macro + discovery` commit (see `093cc68`).

**Per-milestone deliverables checklist** (mirrors `5418614`'s file list):

- `Sources/ProtocolLawKit/Public/<Proto>Laws.swift` (one file per protocol)
- `Sources/ProtoLawCore/KnownProtocol.swift` — append cases + map entries + switch arms
- `Sources/ProtoLawDiscoveryTool/AdvisorySuggester.swift` — only if a syntactic detector makes sense (probably skip for the numeric cluster — false-positive rate would blow past the 5% bar)
- `Sources/ProtocolLawKit/Model/LawIdentifier.swift` — append `enum <Proto>Law` cases + factory methods
- `Tests/ProtocolLawKitTests/<Proto>LawsTests.swift` — runs against `Int`, `UInt`, `Int32`, `Double` etc.
- `Tests/ProtocolLawKitTests/PlantedBugs/Planted<Proto>Violators.swift`
- `Tests/ProtocolLawKitTests/PlantedBugs/PlantedBug<Proto>DetectionTests.swift`
- `Tests/ProtoLawMacroTests/MacroExpansionTests.swift` + `EmitterGoldenTests.swift`
- `docs/Protocols/SwiftProtocolLaws PRD.md` §4.3 — add a section per protocol, drop from v1.1+ candidates list

## 4. Generator strategy

Stock `swift-property-based` ships generators for `Int`, `Int8/16/32/64`, `UInt`, `UInt8/16/32/64`, `Float`, `Double`. These cover the entire FixedWidthInteger + BinaryFloatingPoint surface. The check signatures are protocol-generic and accept any `Generator<T, _>`.

Signature shape (mirrors `SetAlgebraLaws.swift:29`):

```swift
public func checkAdditiveArithmeticProtocolLaws<
    Value: AdditiveArithmetic & Equatable & Sendable,
    Shrinker: SendableSequenceType
>(
    for type: Value.Type = Value.self,
    using generator: Generator<Value, Shrinker>,
    options: LawCheckOptions = LawCheckOptions()
) async throws -> [CheckResult]
```

For `Numeric`/`BinaryInteger`/`FixedWidthInteger`, the same shape — `Numeric` already extends `Equatable` (transitively, via Hashable in BinaryInteger; AdditiveArithmetic itself does require Equatable). For `FloatingPoint`/`BinaryFloatingPoint`, add `Equatable` to the bound for symmetry.

**Add `laws: LawSelection = .all` on every refinement** (BinaryInteger, FixedWidthInteger, SignedNumeric, FloatingPoint, BinaryFloatingPoint) — same pattern as `StrideableLaws.swift:28`. The chain calls into a private `collectInherited<Parent>` helper.

**Bounded-generator caveat for multiplication-heavy laws.** `Numeric.multiplicationAssociativity` over `Gen<Int>.int()` (default unbounded) overflows on >50% of triples. Ship convenience wrappers: `Gen<T: FixedWidthInteger>.boundedForArithmetic()` returning values in `±sqrt(T.max)`. Lands in M2 alongside BinaryInteger as `Sources/ProtocolLawKit/Public/NumericGenerators.swift`.

**Float/Double NaN injection.** The existing `Gen<Double>.double()` doesn't produce NaN by default. For `.allowNaN`-gated laws, the kit needs `Gen<Double>.doubleWithNaN()` and `Gen<Float>.floatWithNaN()` as part of M4 — the NaN-gated laws explicitly use these via a separate parameter so callers don't accidentally hit NaN-eq-NaN failures in unrelated FloatingPoint laws.

## 5. NaN gating mechanics

Currently `LawCheckOptions` (at `Sources/ProtocolLawKit/Model/LawCheckOptions.swift:13`) has six fields: `budget`, `enforcement`, `seed`, `suppressions`, `backend`, `expectedReplayEnvironment`, `replayRelaxation`. **It does NOT yet have an `allowNaN` field** — the PRD references it (line 171, 346) but it has not been implemented. M4 adds it.

Recommended shape:

```swift
public var allowNaN: Bool   // default false
```

Place it after `replayRelaxation`. Keep it a `Bool`, not an enum.

**Always-on FloatingPoint laws (run regardless of `allowNaN`):**
- `infinityIsInfinite`, `negativeInfinityComparison`, `zeroIsZero`, `signedZeroEquality`, `roundedZeroIdentity`, `additiveInverseFinite`, `nextUpDownRoundTrip`, `signMatchesIsLessThanZero`.

**Gated by `options.allowNaN == true`:**
- `nanIsNaN`, `nanInequality`, `nanPropagatesAddition`, `nanPropagatesMultiplication`, `nanComparisonIsUnordered`, `absoluteValueNonNegative`.

**Coordination with Equatable.** When `allowNaN` is true and `Value` is `Double`, the inherited `Equatable.reflexivity` check will fail on NaN samples. Equatable's reflexivity check must skip NaN samples when `options.allowNaN == true`. This is the only retroactive change to v1.0 laws — Equatable and Comparable both gain a NaN-skip predicate when `allowNaN` is set. Land this as part of M4 commit `(2/3)`.

## 6. Validation impact

**Pass 1 rescan: minimal new emit-able conformances expected.** App code essentially never declares `: AdditiveArithmetic` / `: Numeric` / `: BinaryInteger` etc. in inheritance clauses — these protocols exist to constrain stdlib generic algorithms; the only types that adopt them are stdlib's own `Int`/`UInt` family.

**Pass 2 expansion is genuinely worth scoping into M5.** Add:

- `Int32` test target — exercises `FixedWidthInteger`, transitively running BinaryInteger + SignedInteger + Numeric + AdditiveArithmetic + Hashable. **First time the kit's own laws actually run on stdlib types** — significant validation milestone.
- `Double` test target — exercises `BinaryFloatingPoint`, with `allowNaN: false` for the default run plus a separate `@Test func` with `allowNaN: true`.
- `UInt` — exercises `FixedWidthInteger & UnsignedInteger`.

Land these in `Validation/Tests/StdlibNumericLawsTests.swift` (new file). Effort is ~30 lines.

## 7. Open questions / risks

1. **Multiplication overflow generator hygiene.** `Numeric.multiplicationAssociativity` is mathematically true but operationally trap-prone for `FixedWidthInteger` types under unbounded random sampling. **Resolution:** add the bounded helper in M2; document the contract in the doc comment of `checkNumericProtocolLaws`.

2. **BinaryFloatingPoint ordering vs FloatingPoint.** Keep them split (M4 then M5) because (a) v1.1 set the precedent of one-protocol-per-milestone, and (b) M5 is the natural home for the Pass 2 expansion.

3. **NaN-domain edge cases.** Use `Self.nan` only — the canonical qNaN. Skip `nan.bitPattern` round-trip; not a law, it's an implementation detail.

4. **Float80 / Float16 platform variance.** Scope Pass 2 to `Float`, `Double`, `Int32`, `UInt`, `Int64`. Skip platform-conditional types.

5. **Custom-precision integer types (BigInt-style).** No action required for v1.4 — the laws are protocol-generic by construction.

6. **`BinaryInteger.words` is platform-stride-dependent.** Drop `wordsRoundTrip` from BinaryInteger; defer to FixedWidthInteger where word semantics are precise.

7. **Diamond inheritance — SignedInteger.** Add a unit test in `ProtoLawCoreTests/` that exercises `mostSpecific(in: [.signedInteger])` and asserts the result is `[.signedInteger]` only.

8. **`Numeric` requires `ExpressibleByIntegerLiteral`.** The planted-bug fixtures need to actually conform. Plan for fixture verbosity in M1.

9. **`FloatingPoint.zero` vs `Self(0)`.** Use `.zero` consistently.

10. **Macro test goldens churn.** Plan for ~30-50 line golden diff per milestone — review-able but worth flagging in PR descriptions.
